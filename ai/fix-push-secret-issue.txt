Here's the same guide in plain text:

---

**Fix 1 — CI/CD Pipeline (Start Here, ~15 minutes)**

Create a file at `.github/workflows/deploy.yml` in your repo. Inside it, set up a workflow that triggers on every push to main and makes a POST request to your Render deploy hook URL. Store that URL as a GitHub secret called `RENDER_DEPLOY_HOOK_URL` — you get it from Render's dashboard under Settings → Deploy Hook. That's it. Every push to main now auto-deploys.

---

**Fix 2 — Async Processing (~1 hour)**

Right now your controller blocks until the entire pipeline finishes, which risks timeouts. The quick fix in Spring Boot is to add `@EnableAsync` to your main application class, then annotate your processing method with `@Async` and have it return a `CompletableFuture`. Your controller then immediately returns a `202 Accepted` response telling the user their PDF is being generated, while the actual work happens in a background thread. No Redis or extra infrastructure needed for this step.

If traffic grows later and you need proper queue isolation, you can add Redis as a Docker sidecar and use Spring Batch or Spring Integration to replace the `@Async` approach — but that's a Phase 2 concern.

---

**Fix 3 — Email Notification (~1-2 hours, do this alongside Fix 2)**

Add `spring-boot-starter-mail` to your pom.xml. Configure your SMTP credentials in `application.properties` using environment variables so secrets stay out of your repo. Create an `EmailService` bean that uses `JavaMailSender` to send a simple message with the download link. When your async job from Fix 2 finishes generating the PDF, save it somewhere accessible (a temp path, S3, or Cloudflare R2), build a download URL, and call your email service. The user gets an email instead of waiting on the page.

---

**The logical order is:** CI/CD first so deployments are easy, then async so you remove the timeout risk, then email because it naturally completes the async flow. Each fix builds on the previous one.
